1. TL;DR

chdkptp is a small C core (~11k lines) wrapped in a much larger Lua application (~20k lines). The C side does USB transport, PTP framing, and a single vendor opcode (0x9999) multiplexed into 13 CHDK sub-commands. The Lua side is the CLI, the GUI, helper utilities, and a library of Lua snippets that get shipped to the camera for execution.

A Rust port is very feasible. The protocol surface is small and stable. The C transport/protocol layer is roughly 1500–2000 lines of focused logic that translates cleanly to Rust. The host-side Lua can be entirely replaced with idiomatic Rust APIs. The on-camera Lua, however, is a fact of the protocol — see below.

Recommended approach: single crate chdkptp, built on nusb (pure-Rust USB), async-first API. Port the PTP and CHDK layers cleanly; bundle the essential camera-side Lua snippets as &'static str constants; ship v0.1 with file ops + script messaging, defer GUI/CLI/live-view to later phases.

2. The truth about Lua — what actually goes away

Critical architectural point

Lua exists in two places in this system. The host-side Lua — what you type at the chdkptp CLI — is what you're (rightly) annoyed by, and it goes away completely. The camera-side Lua interpreter lives inside CHDK firmware itself and is the only way to drive non-trivial camera behavior. It does not go away. The Rust crate will still compose Lua source and ship it to the camera via PTP_CHDK_ExecuteScript; the difference is that you never have to touch Lua — the crate handles it behind typed APIs like cam.list_directory() or cam.shoot().

For power users who do want to write custom Lua snippets to run on the camera, the crate exposes execute_script(source: &str) raw. That's the same escape hatch chdkptp has — it just isn't the default user experience anymore.

4. CHDK PTP sub-commands reference

All multiplexed through PTP_OC_CHDK = 0x9999. Sub-command goes in Param1. Wire envelope is vanilla PTP — no custom framing. Current protocol version is 2.8.

Sub-commandDirectionPurposePhase
Version (0)reqReturn CHDK protocol major/minor version2
ExecuteScriptreq+dataSend Lua / uBASIC source; returns script-id + initial status3
ScriptStatusreqBitmask: RUN, MSG-pending3
ReadScriptMsgreq+dataPull one message (NONE / ERR / RET / USER × value-type)3
WriteScriptMsgreq+dataPush a string into the camera-side script's queue3
ScriptSupportreqWhich interpreters are compiled into this CHDK build3
UploadFilereq+data4-byte filename length, filename, then bytes2
DownloadFilereq+dataPreceded by TempData with the filename2
TempDatareq+dataStash a buffer in scratch (filename for DownloadFile)2
GetDisplayDatareq+dataLive view: lv_data_header + viewport / bitmap / palette4
RemoteCaptureIsReadyreqPoll for capture-over-USB readiness; returns format bitmask4
RemoteCaptureGetDatareq+dataPull JPG / RAW / DNG-header chunk4
GetMemory / SetMemory / CallFunctionreq(+data)Read/write camera RAM, call arbitrary function pointer5+

5. Existing Rust ecosystem

Build on

  • nusb — pure-Rust USB, async-first, no libusb dep, actively maintained. Recommended transport.
  • rawloader — if/when we want DNG/RAW; explicitly handles CHDK-produced raws.
  • thiserror, bytes, byteorder for ergonomics. Maybe tokio for the async runtime in examples.

Reference, don't depend

  • ptp crate (3drobotics, 2018) — small, has the right shape (command(code: u16, ...) takes any opcode incl. vendor), but pinned to unmaintained libusb 0.3. Useful as a reading reference; not a dependency.
  • gphoto2-rs — libgphoto2 wrapper. Doesn't expose vendor opcodes, no CHDK awareness. Skip.
  • mtp-rs — newer, async, exposes low-level PTP. Worth a look but probably more abstraction than we want.

Existing CHDK-in-Rust?

None. Crates.io and GitHub topic search both empty. We're filling a real gap.

6. Proposed Rust architecture

One crate, chdkptp, with internal modules. Workspaces add ceremony without helping users — cargo add chdkptp is the goal. We can split later if there's appetite.

7. API sketch

What it should feel like at the call site:

use chdkptp::{Camera, ScriptValue, ScriptMsg};

#[tokio::main]
async fn main() -> chdkptp::Result<()> {
    // Discovery
    let devices = chdkptp::list_cameras()?;
    let mut cam = Camera::connect(&devices[0]).await?;

    // Handshake
    let chdk_ver = cam.chdk_version().await?;
    println!("CHDK protocol {}.{}", chdk_ver.major, chdk_ver.minor);

    // High-level: bundled rlib does the heavy lifting under the hood
    let entries = cam.list_directory("A/DCIM/100CANON").await?;
    for e in entries {
        println!("{}  {} bytes", e.name, e.size);
    }

    // Raw escape hatch: ship arbitrary Lua to the camera
    let id = cam.execute_script("return get_zoom(), get_focus()").await?;
    while let Some(msg) = cam.read_script_msg().await? {
        match msg {
            ScriptMsg::Return { value: ScriptValue::Table(s), .. } => {
                println!("zoom+focus = {s}");
                break;
            }
            ScriptMsg::Error { text, .. } => eprintln!("camera err: {text}"),
            ScriptMsg::User { .. } | ScriptMsg::None => {}
            _ => {}
        }
    }

    // File transfer
    cam.download_file("A/DCIM/100CANON/IMG_0001.JPG", "img.jpg").await?;
    Ok(())
}

PHASE 1 doneactual: ~30 min

Outcome: byte-identical round-trip of a small text file, plus end-to-end shoot_and_download example that takes a picture, derives the new file's camera-side path via get_image_dir() + get_exp_count(), and pulls the 5.5 MB JPEG to local disk at 4.75 MB/s.

Outcome: picture taken — get_exp_count() went 240 → 241 in a single transaction. Plus multi-camera synchronized shoot via thread-per-camera + Barrier.

PHASE 4a doneactual: ~3 h incl. format hunt

PHASE 4b~1–2 days

RemoteCaptureIsReady / GetData for USB-direct stills (JPG / RAW / DNG header chunks). Bypasses the SD card. Optional for tethered shooting workflows. Not yet built.

PHASE 5~2 days
Examples + docs + a thin CLI

Polished examples/, doctested API docs, a small chdkptp-cli binary that covers the 80% case (connect, ls, download, shoot, exec). Not the full chdkptp REPL — just the table-stakes.

PTP/IP transport (ptpip feature). DNG writer via rawloader (dng feature). GetMemory/SetMemory/CallFunction debug ops. Property name tables for diagnostics. uBASIC support (low priority — almost no one writes uBASIC anymore).

9. Findings from hardware testing

Things that came out of bringing the protocol up against a real camera that weren't in the original inventory:

Protocol quirk: ExecuteScript source must be NUL-terminated

CHDK's camera-side handler loads scripts via luaL_loadstring, not luaL_loadbuffer(buf, len). The container length is not used to bound the source — the camera walks the buffer until it hits a \0. Without a trailing NUL, your script gets concatenated with stale bytes from whichever ExecuteScript ran before it, producing baffling parse errors like '<eof>' expected near 'om'. Symptom diagnosis: send "return 42" twice after a longer script; the error mentions bytes from the previous source. Fix is one line in execute_script_lua — append a 0 byte before sending.

  • CHDK protocol version is 2.9 on current builds (the original inventory said 2.8). One minor bump, no breaking changes.
  • Camera defaults to playback mode when USB-attached. shoot() silently no-ops if the camera is in playback — no error, no message, just nothing. The shoot() helper now runs if not get_mode() then switch_mode_usb(1) sleep(2000) end before firing.
  • Multi-valued Lua returns become multiple ScriptMsg::Return messages. get_mode() returns (record_bool, movie_bool, mode_code), which arrives as three separate RET messages in order. Our typed enum doesn't aggregate them — the caller iterates the vec. This is fine; just worth knowing.
  • Camera sleeps off the USB bus after ~30 s of idle. Both list_cameras() and the OS-level ioreg stop seeing the device. Wake by half-pressing shutter or power-cycle. We surface this as Error::NoDevicesFound; long-running tools will need to handle re-enumeration.
  • macOS works without special USB drivers. nusb opens and claims the still-image interface with no entitlements, kexts, or sudo. (The earlier worry about sandboxing turned out to be specific to system_profiler's SPI; ioreg and nusb both work fine.)
  • Time-to-first-pixel for a Rust client is dramatically faster than estimated. Phase 0 + 1 + 2-partial + 3 collapsed from "~2 working weeks for v0.1" to under two hours of focused work, mostly because nusb's API was very well-shaped and the CHDK protocol surface really is as small as advertised.
  • Multi-camera works out of the box. No interface-claim conflicts, no bus contention. Synchronized shoot using one thread per camera + std::sync::Barrier fires within scheduler jitter (sub-ms) on the host side; remaining timing variance is per-camera autofocus.
  • CHDK Lua's filesystem API varies wildly by build. On the ELPH 180 build, os.dir and os.listfiles are both absent; os.stat exists. So generic directory walking doesn't work portably. Workaround: derive the just-taken-photo path from get_image_dir() + get_exp_count() using Canon's deterministic naming (IMG_NNNN.JPG). This is also faster than walking the tree.
  • File transfer throughput: ~4.75 MB/s for IN data phase on USB 2.0 to the ELPH 180 with a 16 KB READ_CHUNK. A 5.5 MB JPEG transfers in 1.16 s. Bumping the chunk to 64 KB may help; the obvious next benchmark.
  • Camera-side Lua propcase numbers are model-specific. On the ELPH 180 build, propcase.FLASH_MODE = 151. Don't hardcode — use pcall(require, 'propcase') to load the camera's own lookup table, then index by name. Falls back gracefully on builds without the module.
  • Camera-side button-event handling differs across firmware revisions. Three identical ELPH 180 units in the bench behaved differently: two would fire from press('shoot_full') after a held half-press, one would not (heard the AF chime but no shutter). Root cause: low-level press/release is not portable across all CHDK builds; CHDK's high-level shoot() Lua function is the only reliably portable trigger. Tight-sync use cases hit a real reliability/precision tradeoff.
  • CHDK auto-releases held keys when a lua_State is destroyed. Verified by probe: press('shoot_half') in one ExecuteScript followed by return get_shooting() in a separate ExecuteScript returns false — even though no release() was called. Implication: any sequence that requires a button to stay held (e.g. pre-locked AF for synchronized fire) must be in a single Lua script. Splitting "warmup" and "fire" across two ExecuteScript calls silently drops the half-press.
  • Canon's menu flash-mode setting affects more than just whether the flash fires. When the menu was set to no flash, the camera engaged a separate low-light auto-capture / scene-detection behavior during long half-press holds — firing the shutter on its own ~2–3 s into the busy-wait, regardless of our set_prop(p.FLASH_MODE, 1) override. Setting the menu to auto flash restored the normal "wait for explicit full-press" interlock and eliminated the stray captures. Lesson: set_prop(p.FLASH_MODE, …) controls the firing policy, not the system-enabled mode bit Canon's auto-capture heuristics check — those live in a different propcase that CHDK doesn't expose by name on this build. Bench-test cameras should be left with menu flash set to auto, not off.
  • Three concurrent bugs in our initial live view decoder, all hiding behind a single "wrong colors" symptom. Each one masked the others. (a) LV_FB_YUV8 = 0 in canonical CHDK enum order, not 1 as I'd guessed — so the descriptor's fb_type=0 was the correct YUV8 marker, not "no data." (b) LV_TFR_BITMAP = 0x04 and LV_TFR_PALETTE = 0x08, not 0x02 and 0x04 — the bitmap-vs-palette flag values are swapped between chdkptp source and what's intuitive from the constant order. (c) The actual viewport format is Y411 / UYVYYY (12 bpp, 6 bytes per 4 pixels: U Y0 V Y1 Y2 Y3 with one chroma pair shared across 4 luma samples) — not YUV4:2:0 planar or YUV4:2:2 packed, both of which happen to fit the same byte count and produce structurally-correct-but-color-wrong images, hiding the real bug. The fix required reading CHDK's liveimg.c directly; the canonical BT.601 coefficients there (integer fixed-point: R += V·5743, G −= U·1411 + V·2925, B += U·7258, all + 2048 >> 12) match nothing else on the internet exactly.
  • The viewfinder pipeline runs only when the LCD is on, and that's controlled by the camera physically (or by user menu setting), not via PTP. Originally I thought we needed to hold shoot_half to "wake the pipeline" before get_display_data could return real frames. Tested at ~2 s per frame using a 30-second held-half-press arm script. Then user pointed out that the LCD is always on on his cameras, so the pipeline is always running. Confirmed: removing the arm logic entirely dropped latency from ~2 s to ~30 ms per frame with no half-press at all. Lesson: don't fight the camera; if the LCD viewfinder is showing pixels, those same pixels are flowing into the YUV buffer we read.
  • macOS ptpcamerad races us for the USB device claim on every connect. Symptom: claim_interface(0): could not be opened for exclusive access. Workarounds: (1) sudo killall ptpcamerad (one-shot, daemon respawns), (2) permanent fix via Image Capture.app → select camera → "Connecting this camera opens: No application" (persists per camera serial). We added retry-with-backoff in chdkpano-server's get_or_open, but the cleanest path is to take the device out of macOS's auto-mount list.

11. Risks & unknowns

  • Hardware testing. resolved — PowerShot ELPH 180 ×2 on the bench, both confirmed working with CHDK 2.9.
  • USB quirks on macOS. resolved — nusb works clean. Linux variant remains untested but the pure-Rust backend doesn't share libusb's URB-flakiness risk.
  • USB quirks at scale. Large file transfers (multi-MB JPEGs, raw dumps) untested. The 16 KB read-chunk size in session.rs should scale, but worth a benchmark when phase 2 lands.
  • Camera firmware variance. Still real. The ELPH 180 doesn't have os.listfiles (it's not part of CHDK Lua core; only some builds), so the directory-walking approach for "find the newest image" needs care. Probe with os.dir instead.
  • UTF-16 string edge cases. resolved — codec verified on a 50-op DeviceInfo dataset and round-trips the camera's Unicode-clean strings.
  • Bundled rlibs version drift. Still relevant for whenever we adopt them. Mitigation unchanged: small, conservative selection; expose execute_script_lua() as the escape hatch (already done).
  • Camera USB sleep. (new) Cameras drop off USB after ~30 s idle. Any long-running tool needs to handle NoDevicesFound by re-enumerating. Worth a future Camera::reconnect().
  • Lua-on-camera is unavoidable. Worth restating: if anyone reads this plan and expects the Rust crate to mean "no Lua anywhere," they'll be surprised. Now reinforced — even the in-memory shoot helper sends a 4-line Lua snippet.
