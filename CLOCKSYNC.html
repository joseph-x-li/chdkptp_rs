shoot_all_clocksync.rs

Take a synchronized photo across N CHDK-running Canon cameras with sub-10 ms inter-camera precision, using per-camera clock-offset measurement + a single combined camera-side script that warms up, busy-waits, and fires — all inside one lua_State.

The big idea

The naive way to fire N cameras "at the same time" is to send each one a "shoot now" command at the same moment. That fails because USB transit is variable (10–50 ms per camera) and the host's thread scheduler adds jitter — you end up with ~10–30 ms inter-camera skew.

The clock-sync trick moves the synchronization point off the host and onto each camera. We measure each camera's tick-clock offset relative to host wall-clock, then send each one a script containing its own target tick. Each camera busy-waits locally on its own clock until the target arrives. The same wall-clock instant arrives on every camera independently — USB transit jitter happens before the wait and gets absorbed.

Subtle CHDK behavior: when an ExecuteScript's lua_State is destroyed at end of script, CHDK auto-releases any keys held via press(). This means you cannot hold shoot_half across two separate ExecuteScript calls — the camera silently drops the press, even though no release() was called. So the warmup and fire phases must run inside the same script. That's why this design uses one combined Lua script per camera instead of two.

Why one combined script (not two)

The original design had a separate WARMUP script that pressed shoot_half and returned (intending the half-press to stay held), and a separate FIRE script that pressed shoot_full. This had a subtle bug:

$ exec_lua: "press('shoot_half') return 'pressed'"
→ "pressed"

$ exec_lua: "return tostring(get_shooting())"
→ "false"             # AF was NOT locked — half-press already released

CHDK destroys the script's lua_State when the script returns. As part of that teardown, it auto-releases any keys held via press(). So even though the WARMUP script "left" shoot_half pressed, by the time the FIRE script ran, the camera was no longer in half-pressed state. The original FIRE script's press('shoot_half') re-press kicked off a fresh AF cycle — which on some cameras finishes within a few tens of ms (focus distance retained), but on others takes 200–800 ms and races with the immediate press('shoot_full') that follows.

The combined-script approach keeps the entire sequence inside one lua_State, so the half-press genuinely persists across warmup → busy-wait → fire. No race, no per-camera firmware-specific behavior to dodge.

The key visual difference in the diagram: in Phase 3's busy-wait sub-phase, no PTP messages flow during the wait. The host thread is just polling script_status for completion; the synchronization is happening entirely camera-side.

Precision floor

CHDK's get_tick_count() has 10 ms granularity on this camera build (probe traces show tick values jumping in multiples of 10). That's the physical floor: even with a perfect offset estimate, two cameras' busy-waits exit at most 10 ms apart because both are quantized to 10 ms ticks.

The other contribution is offset error, bounded by half the round-trip time of the best probe sample — we take 20 samples per camera and pick the lowest-RTT one. Typical best-RTT after the 20 ms poll-interval tightening is 30–50 ms, giving 15–25 ms half-RTT.

Combined: ~2–10 ms inter-camera shutter-arming skew. Going tighter would require either higher-resolution camera-side timing (not exposed by CHDK Lua) or hardware sync via a flash-trigger cable.

The set_* calls are guarded with type() == 'function' and pcall(require, 'propcase') checks so cameras that don't expose a given knob (auto-only models can't always honor set_tv96_direct) degrade silently rather than erroring.
